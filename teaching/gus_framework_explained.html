The GUS Framework — Complete Explanation

GUS stands for Graph Utility Scoring. It is the research framework built around the TH-GFD codebase that answers one practical engineering question:

The central question GUS answers:
You have a fraud detection problem. Should you spend the time building a Graph Neural Network, or will a plain XGBoost model give you the same result? How can you know before spending weeks building the GNN?

That is it. Everything in the framework — the models, the datasets, the metrics, the two-phase training — is built in service of answering that question with data.

Part 1 — Why This Problem Exists

The situation

GNNs are expensive. They require graph construction, a different data pipeline, new model code, longer training, and deeper expertise to debug. If XGBoost already gives you AUC = 0.97, spending a month building a GNN to get AUC = 0.98 is almost certainly not worth it.

On the other hand, some fraud problems are fundamentally relational. Fraud rings, shared accounts, coordinated bot networks — these patterns are invisible to tabular models but obvious to a GNN. On those datasets, GNNs can improve AUC by 10–20 points.

The problem is that before GUS, there was no principled way to tell which situation you were in. Researchers would train a GNN, discover it offered no improvement, and write it off as a failed experiment — without understanding why it failed or how to predict failure in advance.

What GUS proposes

GUS proposes that whether a GNN helps depends on two measurable properties of your dataset that you can compute before training any model:

  1. How much can the raw features predict fraud? (called Feature Predictiveness)
    If features alone give near-perfect AUC, there is nothing left for the graph structure to add.
  2. How strongly do fraud nodes cluster together in the graph? (called Fraud Homophily)
    If fraudsters tend to connect to other fraudsters, a GNN can propagate the fraud signal through the graph. If fraud nodes are scattered randomly, message passing will average away the signal.

From these two properties, GUS computes a single number called Graph Utility, and uses it to decide whether to recommend a GNN or a tabular model.

Part 2 — The Three Measurements GUS Makes

Measurement 1: Feature Predictiveness

What it is: Train a simple model (logistic regression or random forest) using only the raw tabular features — no graph at all. Measure the AUC. That is the Feature Predictiveness score.

Why it matters: It tells you how much of the fraud signal is already captured by the features alone. If a transaction amount of $50,000 from country X through declarant Y is already 97% likely to be fraud based on those numbers, the graph adds nothing.

Range: 0.5 (random) to 1.0 (perfect).
Low score (0.5–0.7): features are weak, there is room for the graph to add value.
High score (0.9+): features already explain fraud well, GNN is unlikely to help much. 

# From analyze_datasets.py
# Feature Predictiveness: AUC of a simple model using only features

from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import roc_auc_score

rf = RandomForestClassifier(n_estimators=100)
rf.fit(X_train, y_train)
feature_pred = roc_auc_score(y_test, rf.predict_proba(X_test)[:, 1])

# Example values from the actual experiments:
# Credit Card fraud:   feature_pred = 0.974  (features almost perfect)
# Ecommerce fraud:     feature_pred = 0.751  (features moderate)
# Vehicle Loan:        feature_pred = 0.631  (features weak)

Measurement 2: Fraud Homophily

What it is: Build a graph where transactions that share an entity (same importer, same IP, same device) are connected. Then ask: what fraction of the edges connect two nodes that have the same label (both fraud, or both legitimate)?

Formally: homophily = (edges between same-label nodes) / (total edges)

Why it matters: Message passing in a GNN averages a node's embedding with its neighbors' embeddings. If neighbors tend to share the same label, this averaging strengthens the signal. If neighbors are a random mix of fraud and legitimate, the averaging destroys the signal.

Range: 0 to 1.
Above 0.5 means nodes more often connect to same-class neighbors — useful for GNN.
Below 0.5 means nodes more often connect to opposite-class neighbors — harmful for GNN. 

# From analyze_datasets.py — compute_homophily()

def compute_homophily(edge_index, y):
    src = edge_index[0]
    dst = edge_index[1]
    # What fraction of edges connect nodes with the same label?
    same_label = (y[src] == y[dst]).mean()
    return same_label

# Example values from the actual experiments:
# IP Blocklist:    fraud_homo = 0.777  (fraud nodes cluster tightly)
# Ecommerce:       fraud_homo = 0.734  (good clustering)
# Vehicle Loan:    fraud_homo = 0.238  (fraud is scattered)
# Credit Card:     fraud_homo = 0.041  (fraud is nearly isolated)

Measurement 3: Graph Utility and Effective Graph Utility

Graph Utility is a raw score indicating how much value the graph structure adds beyond the features. The framework computes it empirically: train both a tabular model and a GNN, subtract their AUC scores. 

Graph_Utility = AUC(GNN) - AUC(best_tabular_model)

But Graph Utility alone is misleading. A dataset where XGBoost already scores AUC = 0.9997 cannot be improved by much — there simply is no room. So GUS introduces Effective Graph Utility (EGU): 

# From complete_gnn_experiments.py — DatasetSpec

@property
def effective_gu(self) -> float:
    # Scales Graph_Utility by how much room for improvement exists
    return self.graph_utility * (1 - self.xgb_auc)

# Examples:
# IP Blocklist:   GU=0.2895, XGB_AUC=0.9116  ->  EGU = 0.2895 * (1 - 0.9116) = 0.0256
# Customs:        GU=0.11,   XGB_AUC=0.9997  ->  EGU = 0.11   * (1 - 0.9997) = 0.000033
# Ecommerce:      GU=0.06,   XGB_AUC=0.7808  ->  EGU = 0.06   * (1 - 0.7808) = 0.0132
The key insight: Customs Fraud has a Graph_Utility of 0.11 — which sounds meaningful — but an Effective_GU of only 0.000033, because XGBoost already scores 0.9997. There is nothing left to improve. The GUS framework catches this and correctly recommends XGBoost for Customs, even though the raw graph utility looks non-zero.

Part 3 — The Decision Rule

After measuring EGU, the framework applies a simple threshold rule: 

IF Effective_GU > 0.01:
    RECOMMEND GNN
ELSE:
    RECOMMEND XGBoost / tabular model

This rule was validated on 12 real-world fraud datasets and achieved 75% accuracy in predicting which model type would win. The threshold 0.01 was determined empirically by finding the value that best separates datasets where GNNs improved performance from those where they did not.

Part 4 — The AUC Prediction Formula

GUS goes further. It builds a regression model that predicts what AUC a baseline model will achieve on a new dataset, using only the graph structural properties. This lets you estimate model performance before training. 

# From EXPERIMENT_RESULTS.md — linear regression fit on 12 datasets

Predicted_AUC = 0.611 × Feature_Pred
             +  (-0.038) × Fraud_Homo
             +  0.076 × Fraud_Rate
             +  0.398

R² = 0.6095

Reading this formula:

  • Feature Predictiveness (coefficient 0.611, strongest): The single most important predictor of AUC. If features are strong, AUC will be high.
  • Fraud Homophily (coefficient -0.038, small and negative): High homophily actually slightly lowers baseline AUC in this formula. This makes sense: high homophily means fraud clusters together, which makes it easier to detect with a GNN but may make the raw features less individually discriminative (individual transactions look similar to their neighbors).
  • Fraud Rate (coefficient 0.076, small positive): Slightly higher fraud rate makes AUC slightly higher, probably because there are more positive examples to learn from.
R² = 0.61 means this formula explains about 61% of AUC variance across datasets. It is not a perfect predictor, but it is useful for a quick estimate on a new dataset before running any experiments.

Part 5 — The TH-GFD Model (The GNN That GUS Tests)

When GUS recommends a GNN, the specific model it uses is TH-GFD. Understanding TH-GFD is understanding what GUS deploys when it says "a GNN will help here."

TH-GFD has four distinguishing design choices that make it well-suited for fraud detection specifically:

Design Choice 1 — Heterogeneous graph (multiple node types)

Rather than a single type of node, TH-GFD builds a graph with different kinds of nodes — transactions, importers, declarants, countries, offices. Each transaction is connected to the entities it shares. Two transactions are linked if and only if they share an entity.

Why this matters: it models the real structure of fraud. A fraud ring is not a set of transactions with similar amounts. It is a set of transactions that share an importer or a customs agent who is coordinating the evasion. The heterogeneous graph captures exactly that structure.

HETEROGENEOUS GRAPH STRUCTURE ------------------------------ T001 ---[to_importer]---> IMP_A T002 ---[to_importer]---> IMP_A <- T001 and T002 share IMP_A T003 ---[to_importer]---> IMP_B and are therefore indirectly connected T001 ---[to_declarant]-> DEC_07 T004 ---[to_declarant]-> DEC_07 <- T001 and T004 share DEC_07 T001 ---[to_country]---> CN T002 ---[to_country]---> CN Node types: {transaction, importer, declarant, office, country} Edge types: {tx->importer, tx->declarant, tx->office, tx->country} + all reverse edges {importer->tx, declarant->tx, ...}

Design Choice 2 — Temporal decay (time-aware edges)

Every edge carries the timestamp of when the transaction occurred. When computing attention weights during message passing, edges from the distant past get lower weight than recent edges. The decay formula is: 

phi(delta_t) = exp( -lambda * delta_t )

delta_t  = current_time - edge_time   (days ago)
lambda   = learnable decay rate, one per relation type
           (importers might decay slowly, countries might decay fast)

Result:  a weight in [0, 1]
         edge from yesterday  ->  weight near 1.0
         edge from 3 years ago -> weight near 0.0

Why this matters: fraud patterns change. An importer who was fraudulent in 2018 may have reformed, or may have been caught and replaced. Treating old and new connections with equal weight would degrade predictions.

Design Choice 3 — Relation-specific attention (different weights per edge type)

Not all connections are equally informative. Sharing the same importer might be a stronger fraud signal than sharing the same country of origin. TH-GFD learns a separate attention weight for each relation type. 

# For each edge type (e.g., importer edges):
attention_score = LeakyReLU( W_importer * [h_src || h_dst] )
                  * temporal_decay(delta_t)

# Softmax over all importer neighbors of this transaction
# -> model learns how much to trust each individual importer connection

Design Choice 4 — Two-head output (fraud + revenue)

The model predicts two things simultaneously from the same embedding: whether a transaction is fraud, and how much revenue would be recovered if it were inspected. This is relevant for customs agencies that must prioritize inspections to maximize recovered tax revenue, not just fraud count. 

tx_embedding  (64-dim, one per transaction)
     |
     +---> classifier head ---> fraud_probability (0 to 1)
     |
     +---> revenue head    ---> expected_revenue ($)

Loss = cross_entropy(fraud_pred, fraud_label)
     + 0.3 * MSE(revenue_pred, actual_revenue)

Part 6 — Two-Phase Training (Why and How)

This is one of the most important parts of GUS/TH-GFD and one that is easiest to misunderstand. The training has two completely separate phases. Understanding why requires understanding the label scarcity problem.

The label scarcity problem

In customs fraud detection, a shipment only gets a label (fraud / not fraud) if it is physically inspected. Agencies inspect roughly 1–10% of shipments. The other 90–99% have no label at all — we do not know if they are fraud.

This means you cannot simply train a supervised classifier, because 90% of your data has no supervision signal. You need a way to extract useful information from the unlabeled data.

Phase 1 — Self-supervised pre-training (no labels required)

Before using any fraud labels, the model learns useful representations by solving a self-supervised task. The task is: given two different "views" of the same transaction (created by randomly perturbing the graph), the model must produce similar embeddings for both views.

Two types of perturbation are used: 

# From experiments/train_thgfd.py — THGFDTrainer

# View 1: Temporal augmentation
# Drop some old edges, keep mostly recent ones
# -> simulates seeing the transaction from a "recent history" perspective
data_aug1 = temporal_augmentation(data)

# View 2: Relation augmentation
# Randomly mask out one entire relation type (e.g., hide all country edges)
# -> simulates seeing the transaction without one type of context
data_aug2 = relation_augmentation(data)

# Get embeddings for both views
z1 = model.get_embeddings(data_aug1)   # (N_transactions, 64)
z2 = model.get_embeddings(data_aug2)   # (N_transactions, 64)

# Contrastive loss (NT-Xent):
# - Same transaction in both views -> push their embeddings CLOSE together
# - Different transactions           -> push their embeddings FAR apart
loss_pretrain = contrastive_loss(z1, z2)

The contrastive loss formula: 

z1 = normalize(z1)   # L2 normalize to unit sphere
z2 = normalize(z2)

similarity_matrix[i][j] = dot(z1[i], z2[j]) / temperature
# temperature = 0.5 (controls sharpness)

# For transaction i:
#   positive pair: (z1[i], z2[i]) -- same transaction, different view
#   negative pairs: (z1[i], z2[j]) for all j != i
loss[i] = -similarity_matrix[i][i]
        + log( sum( exp(similarity_matrix[i][j]) ) for all j )

After pre-training, the model has learned that transactions with similar entities and similar time patterns should have similar embeddings — without ever seeing a fraud label.

Phase 2 — Semi-supervised fine-tuning (with labels + pseudo-labels)

Now we switch to supervised learning, using only the small set of labeled transactions. Three loss terms are combined: 

# Term 1: Supervised fraud loss (only on labeled transactions)
loss_fraud = cross_entropy(fraud_logits[labeled_mask], y[labeled_mask])

# Term 2: Revenue prediction loss (only on labeled transactions)
loss_revenue = MSE(revenue_pred[labeled_mask], actual_revenue[labeled_mask])

# Term 3: Pseudo-label loss (on unlabeled transactions)
# If the model is very confident about an unlabeled transaction,
# treat that prediction as ground truth and train on it

fraud_proba = sigmoid(fraud_logits[unlabeled_mask])

pseudo_fraud = (fraud_proba > threshold)     # model thinks this is fraud
pseudo_legit = (fraud_proba < 1 - threshold) # model thinks this is legitimate

loss_pseudo = cross_entropy(logits[pseudo_fraud], ones)
            + cross_entropy(logits[pseudo_legit], zeros)

# Curriculum: threshold starts strict (0.95) and gradually relaxes to 0.70
# This means we only add easy pseudo-labels first, then harder ones later
threshold = 0.95 -> 0.70   (decreases over training epochs)

# Total loss
loss = loss_fraud + 0.3 * loss_revenue + 0.1 * loss_pseudo
Why curriculum? Starting with threshold = 0.95 means only the most obvious frauds get pseudo-labeled first. As training progresses and the model gets better, we lower the threshold to 0.70 and include more borderline cases. This prevents the model from confidently pseudo-labeling things wrong early in training and then reinforcing those mistakes.

Part 8 — The Evaluation Metrics GUS Uses

Standard accuracy is useless for fraud detection because the classes are extremely imbalanced (often 0.1–5% fraud). GUS uses metrics that measure ranking quality and operational performance.

AUC-ROC

Probability that a randomly chosen fraud case gets a higher score than a randomly chosen legitimate case. Threshold-independent. The primary metric for model selection. 

AUC = P( score(fraud) > score(legitimate) )
0.5 = random guessing
1.0 = perfect ranking

Average Precision (AP)

Area under the precision-recall curve. More sensitive than AUC when the fraud rate is very low, because it focuses on the positive class.

Precision@K%, Recall@K%, Lift@K%

These are operational metrics. In practice, a fraud team can only investigate K% of all transactions per day. These metrics ask: if you investigate the top K% by fraud score, how many are actually fraud? 

K = 1%, 5%, or 10%  (the framework tests all three)

P@K%  = (fraud found in top K%) / K
R@K%  = (fraud found in top K%) / (total fraud)
Lift  = P@K% / base_fraud_rate   (how much better than random inspection)

Example with K=1%, fraud_rate=5%:
  Lift = 10 means top 1% contains 10x more fraud than a random 1% would.

NDCG

Normalized Discounted Cumulative Gain. Measures ranking quality with more credit for finding fraud at the very top of the list, decreasing credit as you go further down.

Part 9 — The Graphs GUS Builds for Baseline GNNs

When running the GCN/GraphSAGE/GAT baselines, GUS does not use the domain-specific heterogeneous graph (that is only for TH-GFD). Instead it builds a generic graph from the feature space using one of three methods:

Method 1 — KNN Graph (default)

Connect each transaction to its K nearest neighbors in feature space (using Euclidean distance after normalization). K = 10 by default. Used when you do not have domain knowledge about which transactions should be connected. 

# From complete_gnn_experiments.py — create_knn_graph()
nn = NearestNeighbors(n_neighbors=10)
nn.fit(X_scaled)
distances, indices = nn.kneighbors(X_scaled)
# Connect every transaction to its 10 most similar transactions by features
edge_index = [(i, j) for i in range(n) for j in indices[i][1:]]

Method 2 — Similarity Graph

Connect transactions if their cosine similarity exceeds a threshold (0.9). Results in a sparser graph where only very similar transactions are linked.

Method 3 — Label Propagation Graph

For labeled nodes, bias the connections towards same-class neighbors. This explicitly engineers the graph to have high homophily, boosting GNN performance. Used to measure the upper bound of GNN potential on a dataset.

Part 11 — How To Use GUS on a New Dataset (End to End)

If you have a new fraud detection dataset and want to use the GUS framework:

Step 1 — Compute Feature Predictiveness

from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import roc_auc_score

rf = RandomForestClassifier(n_estimators=100, class_weight='balanced')
rf.fit(X_train, y_train)
feature_pred = roc_auc_score(y_test, rf.predict_proba(X_test)[:, 1])

print(f"Feature Predictiveness: {feature_pred:.4f}")
# If > 0.95 -> stop here. Use XGBoost. You are done.

Step 2 — Build the graph and compute Fraud Homophily

# Build graph: connect transactions that share an entity
# (same IP, same user, same device, same card, whatever makes sense for your domain)

from sklearn.neighbors import NearestNeighbors
import numpy as np

nn = NearestNeighbors(n_neighbors=10)
nn.fit(X_scaled)
_, indices = nn.kneighbors(X_scaled)

src = np.repeat(np.arange(len(X_scaled)), 10)
dst = indices[:, 1:].flatten()

# Compute homophily
same_label = (y[src] == y[dst]).mean()
print(f"Fraud Homophily: {same_label:.4f}")
# If < 0.5 -> graph structure may not help GNN

Step 3 — Estimate Effective Graph Utility

# Quick estimate using the formula from GUS
predicted_auc = 0.611 * feature_pred + (-0.038) * fraud_homo + 0.076 * fraud_rate + 0.398

# Rough graph_utility estimate (requires some domain knowledge or a quick GNN run)
# For a first pass, you can use: graph_utility ~ fraud_homo * 0.2 (rough heuristic)

effective_gu = graph_utility * (1 - xgb_auc)
print(f"Effective Graph Utility: {effective_gu:.4f}")

if effective_gu > 0.01:
    print("RECOMMENDATION: Use a GNN (TH-GFD)")
else:
    print("RECOMMENDATION: Use XGBoost — graph adds little value")

Step 4 — If GNN is recommended, run TH-GFD

# python run_experiment.py --data_path your_data.csv --label_ratio 0.05

# Or directly:
from data.graph_builder import TemporalHeteroGraphBuilder
from models.thgfd import THGFD, create_model
from experiments.train_thgfd import THGFDTrainer

# 1. Build the graph
builder = TemporalHeteroGraphBuilder(
    entity_columns=['importer_id', 'country', 'agent_id'],
    numerical_columns=['amount', 'weight', 'tax']
)
data = builder.build(df)

# 2. Create model
model = create_model(data, config)

# 3. Train (phase 1: pretrain, phase 2: finetune)
trainer = THGFDTrainer(model, device='cpu')
trainer.pretrain(data, epochs=100)
trainer.finetune(data, epochs=200, label_ratio=0.05)

# 4. Get predictions
fraud_logits, revenue_pred = model(data)
fraud_proba = torch.sigmoid(fraud_logits)

The GUS framework is both a tool (use it to decide which model to build) and a research contribution (it proves, with data across 12 datasets, that Effective Graph Utility is a statistically significant predictor of GNN benefit over tabular models in fraud detection).
